# SPECA グラフィックレコーディングシリーズ計画

## 目的

このシリーズは、SPECA をマネージャーや関係者に短時間で説明し、導入判断・認識合わせ・説明責任に使える資料セットにすることを目的とする。

重視すること:

- SPECA の本質を「仕様から監査観点を作り、実装で検証する仕組み」として伝える
- 現場実装の細部より、判断材料・説明軸・誤解しやすい点を優先する
- 仕様、性質、コード、候補脆弱性、レビュー、運用判断のつながりを見える化する
- 1枚ごとに単体でも読めるが、並べると説明ストーリーになる構成にする

## 既存 01〜07 の評価

既存7枚は、画風・配色・手描きグラレコ感・情報密度はおおむね統一できている。特に `reference.png` / `01_overview_manager_focus.png` の画風を基準にした太いマーカー線、雲形囲み、人物、黄色ハイライト、青い結論枠はシリーズの核として使える。

一方で、内容面では 03、05、07 がいずれも「マネージャー向け要点」「説明時の観点」「最終まとめ」に近く、少し重複している。今後さらに枚数を増やす場合は、同じまとめを繰り返すより、以下の不足領域を補う方がよい。

- 実際のパイプライン各段階で何が出力されるのか
- プロパティ生成の良し悪しが、検出力にどう影響するのか
- Phase 03 の「証明試行」が何をしているのか
- Phase 04 のレビューで、なぜ実用性が上がるのか
- 評価結果をマネージャーがどう読むべきか
- 導入時に準備するもの、運用体制、リスク

## 結論

推奨は「既存7枚を画風リファレンスとして使い、全体構成は01から再設計する」こと。

理由:

- 既存7枚は試作としては有効だが、説明順としては概要・まとめが複数回出てくる
- 01〜07をそのまま続けると、後続で深掘り領域を足してもシリーズ全体の流れが散りやすい
- 01から再構成すると、マネージャー向けの導入説明、技術理解、判断材料、運用判断を自然な順番に整理できる

ただし、画像そのものは無駄ではない。既存7枚は、画風・情報密度・日本語量・人物やアイコンの使い方のリファレンスとして使う。

## 推奨する全体構成

全体は 12 枚前後が扱いやすい。マネージャー説明では 1〜5 を短時間説明用、6〜10 を詳細説明用、11〜12 を導入判断用として使う。

### 01. SPECA の全体像

役割: 最初に見せる概要ページ。

伝えること:

- 仕様から性質を作り、コードで検証する監査基盤
- コード起点ではなく仕様起点
- 導入意義、判断ポイント、注意点をひと目で示す

既存リファレンス:

- `01_overview_manager_focus.png`

### 02. 仕様起点監査という発想

役割: SPECA の考え方を従来監査と対比する。

伝えること:

- 従来: コードから怪しい箇所を探す
- SPECA: 仕様から守るべき性質を作る
- 仕様準拠が本質になる領域で効く

絵の軸:

- 左に「コードから探す」、右に「仕様から検証」
- 中央に「発想の転換」
- ドキュメント、コード、虫眼鏡、電球のアイコン

### 03. 基本構造: 仕様、性質、コード、証明

役割: コア概念を一段深く説明する。

伝えること:

- 仕様
- 性質
- コード位置
- 証明試行
- 候補脆弱性

既存リファレンス:

- `02_core_concepts.png`

改善ポイント:

- 「SPECAの基本構造」を明確に中央へ置く
- 性質の4分類は短く見せる

### 04. 6フェーズの流れ

役割: 自動パイプラインの全体を説明する。

伝えること:

- 仕様探索
- サブグラフ抽出
- 性質生成
- コード位置解決
- 性質ごとの監査
- レビュー

絵の軸:

- 道路や川のような流れ
- それぞれのフェーズに小さな出力物アイコンを付ける
- 詳細な実装名は避け、何が進むかを視覚化する

### 05. プロパティ生成が検出力を左右する

役割: マネージャーが押さえるべき最重要論点を説明する。

伝えること:

- SPECA の性能は「どんな性質を作れるか」に強く依存する
- 不変条件、事前条件、事後条件、信頼仮定の違い
- 専門知識を性質として入れると効く

絵の軸:

- 良いプロパティは明るいチェックカード
- 曖昧なプロパティはぼんやりした付箋
- 専門家が付箋を追加して精度を上げる構図

### 06. Phase 03: 証明試行の見方

役割: 「証明」と聞いたときの誤解を避ける。

伝えること:

- 数学的な完全形式検証ではない
- LLM による構造化された証拠構築
- Map、Prove、Stress-Test の3段階
- 破綻点が候補脆弱性になる

絵の軸:

- 主張カードをコードに当てる
- チェックできた道と破綻した道を分ける
- 攻撃シナリオにつながる赤い点線

### 07. Phase 04: レビューと誤検知削減

役割: 候補をどう実用的に絞るかを説明する。

伝えること:

- Dead Code
- Trust Boundary
- Scope Check
- 重大な真陽性を落とさず誤検知を減らす

絵の軸:

- 3つのゲート
- 信号機、盾、スコープ枠
- レビュー後に「使える候補」に絞られる流れ

### 08. トレーサビリティ: なぜ説明しやすいのか

役割: マネージャー向けの価値を明確にする。

伝えること:

- 仕様節
- 性質
- コードパス
- 証明トレース
- 攻撃シナリオ

絵の軸:

- 一本の鎖として表現
- 「なぜその発見なのか」を説明する人物
- 報告・レビュー・意思決定に効くことを示す

### 09. 実績と評価結果の読み方

役割: 数字をただ並べず、何を判断すべきかを示す。

伝えること:

- Sherlock Ethereum Fusaka Audit Contest での H/M/L 回収
- 新規バグ発見
- RepoAudit での precision
- 数字は万能性ではなく、適用可能性と限界を読む材料

絵の軸:

- 実績ボード
- メダルやグラフ風アイコン
- 「どの条件で効いたか」を見る虫眼鏡

注意:

- 細かい数値を詰め込みすぎない
- 「すごい結果」ではなく「判断材料」として見せる

### 10. 導入時に準備するもの

役割: 実際に使う前に何を決めるかを整理する。

伝えること:

- 仕様URL
- 対象リポジトリ
- 対象コミット
- バグバウンティスコープ
- 信頼境界
- 重大度基準
- 検証担当者

絵の軸:

- 導入チェックリスト
- 対象、仕様、スコープ、体制、コストの5項目
- マネージャーが意思決定する構図

### 11. 運用設計: コスト、再開、並列実行

役割: 実務導入の判断材料を示す。

伝えること:

- 並列実行で広く見る
- 部分出力で途中再開できる
- コスト監視と早期停止が重要
- 長い監査を止めずに進める設計

絵の軸:

- 複数ワーカー
- 再開矢印
- 予算メーター
- 出力フォルダ

### 12. マネージャー向け最終まとめ

役割: 説明・判断・共有の最後に使う1枚。

伝えること:

- SPECA は監査を丸投げする道具ではない
- 仕様知識を監査観点に変える仕組み
- 見るべきは「何を検証したか」と「根拠を追えるか」
- 導入判断には、対象、仕様、スコープ、検証体制が必要

既存リファレンス:

- `05_final_takeaways.png`
- `07_explaining_to_stakeholders.png`

## 既存7枚を活かして続ける場合の追加計画

既存01〜07をそのまま使い、08以降を追加するなら、次の順番がよい。

### 08. パイプライン出力の見方

目的:

- 各フェーズが何を出すのかを説明する
- マネージャーが「成果物をどう読むか」を理解できるようにする

主な要素:

- 仕様リスト
- サブグラフ
- 性質リスト
- コード位置
- 監査結果
- レビュー判定

### 09. プロパティ生成の良し悪し

目的:

- 検出力を左右する本丸を説明する

主な要素:

- 良い性質: 具体的、検証可能、スコープ内
- 悪い性質: 曖昧、広すぎる、根拠が薄い
- 専門知識の注入が有効

### 10. 証明試行と攻撃シナリオ

目的:

- Phase 03 の見方を補足する

主な要素:

- 主張を分解する
- コードで満たせるか確認する
- 破綻点を探す
- 攻撃経路に落とす

### 11. レビューゲートの実用性

目的:

- 候補をそのまま使わない理由を説明する

主な要素:

- デッドコード
- 信頼境界
- スコープ
- 重大度調整

### 12. 実績の読み方

目的:

- 評価結果を導入判断の材料にする

主な要素:

- 回収率
- precision
- 新規バグ
- 適用領域
- 限界

### 13. 導入チェックリスト

目的:

- 実務で次に決めることを明確にする

主な要素:

- 仕様はあるか
- 対象コミットは固定したか
- スコープと信頼境界は書けるか
- 検証担当者はいるか
- コストと期間を見積もれるか

最優先リファレンス:

- `graphic-recording/reference.png`

補助リファレンス:

- `graphic-recording/01_overview_manager_focus.png`
- `graphic-recording/02_core_concepts.png`

- `graphic-recording/06_deeper_understanding.png`
- `graphic-recording/07_explaining_to_stakeholders.png`

維持する画風:

- 白または薄い紙の背景
- 太い黒マーカーの見出し
- 赤い見出し下線
- 青、緑、ピンク、オレンジの雲形囲み
- 黄色マーカーの強調
- 太い矢印
- 小さな人物イラスト
- 虫眼鏡、盾、チェック、書類、コード、付箋、信号機、予算メーターなどの手描きアイコン
- 1枚あたり4〜5ブロック程度
- 文字は見出しと短文中心

避ける画風:

- SVG風
- フラットなベクター図
- 無機質なプレゼン図表
- グリッドに整いすぎたレイアウト
- 文字だけの説明資料
- 長文の詰め込み

## 生成時の共通プロンプト方針

各画像では、以下を明示する。

- `reference.png` を最優先の画風リファレンスにする
- 既存画像は内容コピーではなく、シリーズ感の参照にする
- 文字はすべて日本語
- 見出しと短文中心にする
- 手描きライブグラレコ風にする
- SVG、HTML、canvas、mermaid、コード生成風を避ける
- 1枚ごとに単体で読めるが、シリーズ全体の流れにもつながるようにする

## 推奨制作順

1. 01〜12の全体再構成版を作る
2. まず 01〜03 を再作成して画風と粒度を固定する
3. 問題なければ 04〜08 を作る
4. 最後に 09〜12 を作る
5. 全体を並べて、重複・文字量・説明順を調整する

既存7枚を使って継続する場合は、08〜13を追加し、最後に「重複が多いまとめ系ページ」を整理する。
