# SPECA LLM 選定ガイド

> SPECA を動かすために必要な **LLM のレベル感**、**Claude モデルファミリーの使い分け**、**サブスクリプションプランの適用可否**、**コスト見積り**、**実用上の推奨構成** をまとめた独立資料です。
>
> 関連：[セットアップガイド](SPECAセットアップガイド.md) ／ [概略と目次](概略と目次.md)
>
> **本ガイドの範囲**：Claude ファミリー＋ SPECA の RQ2 で実評価された DeepSeek R1。それ以外のモデル（GPT、Gemini、Llama 等）は SPECA での動作検証情報がないため対象外。

## 0. 30 秒サマリ

> **結論**：SPECA は Anthropic の **Claude** を前提に設計されている。実用構成は **Opus 4.x（フェーズ 01a〜01e の仕様理解）＋ Sonnet 4.5（フェーズ 02c〜04 のコード解析）** の使い分け。**Claude Pro / Max のサブスクリプション** で動かせるが、**API 課金より制約が強い**。代替モデルとしては DeepSeek R1 が論文で評価済だが、Claude Sonnet 4.5 の方が高精度で結局得。1 ターゲット規模での目安コストは **$30〜100**、バグあたり **$1.69**（RQ2 RepoAudit）。

## 1. SPECA は LLM をどう使っているか（フェーズ別の負荷）

第 3 章「パイプライン詳細」の繰り返しになりますが、LLM 選定の前提として整理します。

### 1.1 フェーズ別のモデル既定値

`scripts/orchestrator/config.py` で各フェーズに `model` フィールドがあります（指定が無いフェーズは Claude Code CLI の **既定モデル＝Opus 系**を使います）：

| Phase | 既定モデル | 仕事の性質 | 推奨ティア |
|---|---|---|---|
| 01a 仕様文書発見 | Opus（既定） | 概念的・推論的：仕様サイトをクロールして「仕様らしい URL」を判定 | **High（Opus 推奨）** |
| 01b サブグラフ抽出 | Opus（既定） | 形式手法的：仕様文書 → Nielson & Nielson プログラムグラフ | **High（Opus 推奨）** |
| 01e プロパティ生成 | Opus（既定） | 抽象推論：信頼モデル分析 + STRIDE + CWE Top 25 | **High（Opus 推奨）** |
| 02c コード位置事前解決 | **Sonnet** 明示 | 反復的：プロパティ → コード位置の検索 | **Mid（Sonnet）** |
| 03 監査マップ生成 | **Sonnet** 明示 | 重い反復＋深い読解：Map → Prove → Stress-Test | **Mid（Sonnet）** |
| 04 監査レビュー | **Sonnet** 明示 | 機械的判定：3 ゲートフィルタ＋セベリティキャリブレーション | **Mid（Sonnet）** |

### 1.2 仕事の性質による要求レベル

| 性質 | 必要な能力 |
|---|---|
| **概念的・抽象的推論**（仕様→プロパティ） | 高度な抽象化と多段推論。ここをサボると **後段すべての精度が下がる**（Property Adherence Effect） |
| **長文コンテキストの完全読解**（Map 段階） | 最低 100K〜200K トークンのコンテキスト窓。関数全体・呼出元・呼出先を同時保持 |
| **構造化指示への忠実性**（Prove／Stress-Test） | 「サブフェーズを順守する」「ショートカットしない」というプロンプト指示への従順さ |
| **JSON 出力の整形性** | スキーマ違反の少なさ（後段スキーマ検証で warning が増えるとパイプライン全体が脆くなる） |

このうち **「仕様→プロパティ」が最重要**。論文の **Property Adherence Effect**（第 9 章 §9.3.6）で示されたとおり、「**プロパティ生成（フェーズ 01a〜01e）が検出カバレッジを律速する制約**」だからです。

### 1.3 「フェーズ 1〜3 を Opus、4〜6 を Sonnet」の論理

論文・README に明記されている使い分け：

> Phases 1–3 use Claude **Opus** (specification understanding); Phases 4–6 use Claude **Sonnet** (code analysis & review).

これは：

- **Opus**：高い抽象推論力。仕様文書を読み解くのに必要。リクエストあたりコストは高いが、**01a〜01e の総量自体は少ない**（仕様 1 つに対して 1 回ずつ）
- **Sonnet**：コスト効率が圧倒的に良く、コード解析の反復に向く。**フェーズ 03／04 は所見数百件 × 多ターン** なので、ここを Sonnet にしてコスト最適化

このバランスが SPECA のコスト効率（バグあたり $1.69）の根拠です。

## 2. Claude ファミリーの選択肢（2026 年 5 月時点）

Anthropic が公開している主要モデル（4.x 世代）：

| モデル | 特徴 | API 価格（per 1M tok） | コンテキスト窓 |
|---|---|---|---|
| **Opus 4.7** | 最高品質。複雑推論・長文理解の最上位 | input $15 / output $75 | 200K（1M 拡張あり） |
| **Opus 4.6** | 4.7 の前世代。fast モードあり | input $15 / output $75 | 200K |
| **Sonnet 4.6** | 最新の中位モデル | input $3 / output $15 | 200K（1M 拡張あり） |
| **Sonnet 4.5** | SPECA RQ2 で評価された主力モデル | input $3 / output $15 | 200K |
| **Haiku 4.5** | 軽量・低コスト | input $1 / output $5 | 200K |

> **注記**：価格・モデル名は記事執筆時点（2026 年 5 月）。最新は [Anthropic 公式](https://www.anthropic.com/pricing) を確認してください。3.x 世代（Sonnet 3.5/3.7）も論文比較対象として登場しますが、2026 年時点で現役の選択肢としては 4.x 世代が主軸。

### 2.1 SPECA の Claude Code CLI でのモデル指定

`scripts/orchestrator/config.py` で `model="sonnet"` のように **エイリアス** を渡しています。これは Claude Code CLI の `--model` フラグにそのまま渡されます：

```bash
# Claude Code CLI のエイリアス
claude --model sonnet ...   # 最新 Sonnet
claude --model opus ...     # 最新 Opus
claude --model haiku ...    # 最新 Haiku
```

特定バージョンを指定したい場合は完全 ID も使えます：

```bash
claude --model claude-sonnet-4-6 ...
claude --model claude-opus-4-7 ...
```

`model` 指定なしのフェーズ（01a／01b／01e）は Claude Code CLI の **既定モデル**（通常は最新 Opus）が使われます。

### 2.2 各モデルの SPECA での「適不適」

| モデル | フェーズ 01a〜01e（仕様理解） | フェーズ 02c（コード位置） | フェーズ 03（監査） | フェーズ 04（レビュー） |
|---|---|---|---|---|
| **Opus 4.7** | ◎ 最適 | ◯ 過剰 | ◯ 過剰（コスト高） | ◯ 過剰 |
| **Opus 4.6** | ○ 推奨 | ○ 過剰 | △ コスト高 | △ コスト高 |
| **Sonnet 4.6** | △ ぎりぎり | ◎ 推奨 | ◎ 推奨 | ◎ 推奨 |
| **Sonnet 4.5** | △ ぎりぎり | ◎ **論文で実証** | ◎ **論文で実証** | ◎ **論文で実証** |
| **Sonnet 3.7** | ✕ 抽象推論不足 | △ 動くが Sonnet 4.5 より精度低 | △ 動くが Sonnet 4.5 より精度低 | ○ 動く |
| **Haiku 4.5** | ✕ 抽象推論不足 | △ 単純なターゲットでは動く | ✕ Map → Prove → Stress-Test の構造化推論が崩れる | △ 単純な FP フィルタなら動く |

### 2.3 Property Adherence Effect の含意

第 9 章で扱った **Property Adherence Effect**（モデルが改善するほどプロパティ準拠度が上がる）が示すこと：

> **モデルを下げると、プロパティスコープからのドリフトが増え、新規候補が増える代わりに FP も増える。モデルを上げると、プロパティ通りにだけ監査するため精度は上がるが、新規候補は減る。**

実用上の含意：

- **「探索的に新規バグを多く拾いたい」**：1 段階下のモデル（Sonnet 4 / 3.7 等）で広く拾う
- **「精度の高い結果だけ欲しい」**：最新 Sonnet 4.5/4.6 で絞る
- **「両方欲しい」**：複数モデルで並走させて結果を統合（コスト 2 倍だが）

## 3. サブスクリプションプラン vs API 課金

ご質問の重要ポイント — **「定額プランを適用できるか」** について整理します。

### 3.1 Anthropic の課金体系（2026 年 5 月時点）

| プラン | 月額 | 提供範囲 | API キーアクセス |
|---|---|---|---|
| **Free** | 無料 | claude.ai 限定的利用 | ✕ |
| **Pro** | $20/月 | claude.ai 個人利用、Claude Code CLI を **Pro レート** で使用可 | △（限定的、要確認） |
| **Max** | $100/月～ | claude.ai 上位プラン、Claude Code CLI を **Max レート** で使用可 | △ |
| **API（従量）** | 使った分だけ | Anthropic Console の API キー、API 直叩き／Claude Code CLI 両対応 | ◎ |
| **Enterprise** | 個別 | カスタム枠 | ◎ |

> **注**：上記は記事執筆時点（2026 年 5 月）の体系。プラン名称・料金は変更され得るため、最新は [Anthropic 公式](https://www.anthropic.com/pricing) を必ず確認してください。

### 3.2 SPECA でサブスクリプションを使えるか

> **使えます。** SPECA は内部で Claude Code CLI を呼び出すだけなので、Claude Code CLI が認証している方法（API キー / OAuth サブスクリプション）を **そのまま継承** します。

具体的に：

```bash
# 方法 A：API キー経由（従量課金）
export ANTHROPIC_API_KEY="sk-ant-..."
uv run python3 scripts/run_phase.py --target 04

# 方法 B：サブスクリプション経由（OAuth でログイン済み）
claude /login   # 一度だけログイン
# 以後は API キー不要
uv run python3 scripts/run_phase.py --target 04
```

両方とも動きます。`ANTHROPIC_API_KEY` が設定されていればそちらが優先、無ければ Claude Code が保存している OAuth トークンが使われます。

### 3.3 サブスクリプション利用時の制約

サブスクリプション枠で SPECA を回す際は、**以下のレート制限／コスト構造の違い** に注意：

| 観点 | API キー（従量） | サブスクリプション |
|---|---|---|
| **追加課金** | トークン消費に応じて発生 | 月額固定（プラン枠内） |
| **レート制限** | Tier 次第（Tier 1 は厳しい、Tier 4 は緩い） | プラン枠（5h ごとのウィンドウ等、Pro/Max で異なる） |
| **長時間ジョブとの相性** | 良い（Tier さえ上げれば） | プラン枠を使い切ると数時間ストールする |
| **大規模監査（10 ターゲット並列等）** | スケールしやすい | **枠を使い切る可能性が高い** |
| **コスト予測性** | 不確実（消費次第） | 完全に固定 |
| **コスト効率** | 大量利用時に逆転（API の方が安い場合あり） | 中規模〜小規模利用で有利 |

### 3.4 用途別の推奨プラン

| ユースケース | 推奨プラン | 理由 |
|---|---|---|
| 個人で 1〜2 ターゲットを試す | **Pro $20/月** | 1 ヶ月まるまる使えば数ターゲット消化可能。月内枠を使い切ったら待つ |
| 個人で並列・継続的に試す | **Max $100/月～** | 高いレート制限／長時間ジョブ向き |
| チームで定期的な監査を行う | **API キー（Tier 引上げ）** | Tier 4（あるいは Enterprise）で並列度を上げ、コストは見える化 |
| 学術検証（論文再現等） | **API キー** | バッチ実行＋コスト記録が容易 |
| 1 回限りの大規模監査（10 実装等） | **API キー** | サブスクの月枠では収まらない |

### 3.5 サブスクリプション利用時の現実的な進め方

Pro / Max でも **動きはする** が、フェーズ 03（最も重い）で問題が起きやすいので、次の運用を推奨：

1. **小さく始める**：`--workers 1 --max-concurrent 2` で動かす
2. **フェーズ 01a〜02c までを先行**して仕様分析を一旦完了させる（プロパティ生成は仕様コーパスに対して 1 回でよい）
3. **フェーズ 03／04 を分割実行**：プロパティを優先度順に並べて、一日で全部やらず数日に分散
4. **`outputs/*_PARTIAL_*.json` 経由の再開**を活用

実際、第 5 章で扱った **「PARTIAL ファイルが第一級市民」** という設計は、サブスクリプションの月枠制約と相性が良いです。途中で止めて翌週続きから、というワークフローが自然にできます。

### 3.6 注意点：Claude Code CLI と「サブスク経由」の現状

`speca-cli`（第 7 章）の M2 マイルストーンで詳述されているとおり、Anthropic の OAuth フローによる **サブスクリプション経由の inference** は、公式には Claude Code CLI 内部メカニズムとして提供されています。

`claude /login` でログインすると Claude Code CLI が API トークンを保存し、以後の inference にそれを使います。SPECA から Claude Code CLI を呼ぶときも同じトークンが使われます。

> **将来リスク**：Anthropic がこのフローを変更した場合、サブスクリプション経由の SPECA 実行は影響を受けます。API キー利用は影響を受けません。

## 4. 代替モデル：DeepSeek R1（論文評価済）

SPECA は基本 Claude 前提ですが、**`scripts/orchestrator/api_runner.py`** に OpenAI 互換 API ランナがあり、論文の RQ2 では DeepSeek R1 で評価しています。

### 4.1 起動方法

```bash
export ORCHESTRATOR_RUNNER=api
export API_RUNNER_BASE_URL="https://openrouter.ai/api/v1"
export API_RUNNER_API_KEY="sk-or-..."
export API_RUNNER_MODEL="deepseek/deepseek-r1"

uv run python3 scripts/run_phase.py --target 04
```

`ORCHESTRATOR_RUNNER=api` を明示しないと起動しません（既定は `claude`）。

### 4.2 論文での実評価（RQ2 RepoAudit）

| 構成 | TP | FP | Precision | 新規候補 | コスト |
|---|---|---|---|---|---|
| **SPECA（Sonnet 4.5）** | — | 6 | **88.9%** | **12** | **$81.05** |
| **SPECA（Sonnet 4）** | — | 13 | 81.2% | 18 | $100.68 |
| **SPECA（DeepSeek R1）** | — | 15 | 72.7% | 7 | **$93.51** |

※ TP は GT 集合への精度比較対象外（第 9 章 §9.3.3 参照）

### 4.3 DeepSeek R1 を選ぶか

| 観点 | 評価 |
|---|---|
| 精度 | Sonnet 4.5 より低い（72.7% vs 88.9%） |
| 新規候補 | Sonnet 4.5 より少ない（7 vs 12） |
| コスト | Sonnet 4.5 より高い（$93.51 vs $81.05） |
| **総合判断** | **論文の数字を見る限り Sonnet 4.5 の方が常に得**。DeepSeek R1 を選ぶ理由は「Anthropic 依存を避けたい」場合に限られる |

### 4.4 OpenRouter 経由のメリット・デメリット

| メリット | デメリット |
|---|---|
| Anthropic 障害時のフェイルオーバー | OpenRouter は中継業者なのでマージン分高い |
| 複数モデルを 1 つのキーで試せる | 一部モデルでツール呼び出しの実装差で動かない |
| データの流れに第三者が入ることへの懸念 | プライバシー・契約条件を別途確認する必要 |

実用上は **「動作確認・比較研究のため」以外の用途で OpenRouter を選ぶ理由は弱い**。

## 5. コスト見積り（実績ベース）

論文と本資料第 9 章の数字から、用途別のコスト目安を整理します。

### 5.1 1 ターゲット監査のコスト（フル監査、Sonnet 4.5）

論文 RQ1（10 個の Ethereum クライアント）の実測値：

| ターゲット規模 | 例 | フェーズ 03 wall-time | 入力トークン | 出力トークン | コスト |
|---|---|---|---|---|---|
| 小（数千 LOC） | nethermind_fusaka 部分 | ~4 分 | 9K | 224K | ~$30〜60 |
| 中（数万 LOC） | grandine_fusaka | ~4 分 | 35K | 773K | ~$30〜60 |
| 中〜大（10 万 LOC〜） | reth_fusaka、prysm_fusaka | 4〜8 分 | 25〜54K | 0.5M〜1.4M | ~$30〜80 |
| 大（数十万 LOC） | lighthouse_fusaka | ~8 分 | 56K | 1.28M | ~$60〜80 |

**合計（10 実装）：約 $400〜620**。

### 5.2 RQ2（C/C++、Sonnet 4.5）

| 指標 | 値 |
|---|---|
| 15 プロジェクト合計コスト | **$81.05** |
| バグ 1 件あたり | **約 $1.69** |
| プロジェクト 1 件あたり | 約 $5.4 |

C/C++ ターゲットは比較的単純で軽量だったケース。Ethereum クライアントよりコスト効率が良い。

### 5.3 サブスクリプションでカバーできるか（試算）

> **重要**：以下は **2026 年 5 月時点の Pro / Max プラン枠の推定** に基づく試算。実際の枠はプランの仕様変更で変わるため、必ず Anthropic 公式の最新情報を確認してください。

仮に Pro $20/月で「月当たり Sonnet 4.5 のトークン枠を $50 相当」確保できるとした場合：

| 監査内容 | 見積コスト | サブスクで賄えるか |
|---|---|---|
| 小規模ターゲット 1 個 | ~$30 | ◎ Pro で十分 |
| 中規模ターゲット 1 個 | ~$60 | ○ Pro でぎりぎり、Max なら余裕 |
| 大規模ターゲット 1 個 | ~$80 | △ Pro では月をまたぐ／Max が安全 |
| 大規模ターゲット 5 個 | ~$300 | ✕ Max でも厳しい、API 推奨 |
| RQ1 相当（10 実装） | ~$500 | ✕ API 必須 |

**実用的な分岐点**：
- **個人・少数ターゲット** → Pro / Max で十分
- **チーム・複数ターゲット並列** → API キー（要 Tier 引上げ）

### 5.4 開発・デバッグ時のコスト管理

```python
# scripts/orchestrator/config.py の PHASE_CONFIGS 内
"03": PhaseConfig(
    ...
    max_budget_usd=200.0,   # ← この値を一時的に下げる
    ...
)
```

開発時は `max_budget_usd=10.0` 等に下げて **早期に止まる** ようにすると、プロンプト変更ミスで暴走するのを防げます。

### 5.5 コストを抑えるテクニック

| テクニック | 効果 |
|---|---|
| `min_severity="Medium"` で Low プロパティを除外 | 02c／03 のアイテム数削減 |
| プロンプトキャッシュ活用（既定で有効） | 同じターゲットを繰り返す場合 30〜80% カット |
| `--workers` を下げて長時間化させる | レート制限を回避、超過コスト無し |
| フェーズ 01〜02c を別途キャッシュ | 仕様分析は 1 回でよい。ターゲット数が増えても再利用 |
| マニュアルプロパティをフェーズ 01e の出力に差し込む | フェーズ 01a〜01e 全体をスキップ可能 |

## 6. ユースケース別推奨構成

### 6.1 個人エンジニア：自分のお気に入り OSS を試す

| 項目 | 推奨 |
|---|---|
| 認証 | **Claude Pro $20/月**（既に契約していれば追加課金 0） |
| モデル | フェーズ 1〜3：Opus（自動）／フェーズ 02c〜04：Sonnet（自動） |
| 並列度 | `--workers 1 --max-concurrent 2`（Pro レート枠考慮） |
| 想定コスト | 月額のサブスク内に収まる前提で、追加課金 0〜数 $ |
| 想定時間 | 1 ターゲットあたり 30〜60 分 |

### 6.2 セキュリティチーム：定期監査

| 項目 | 推奨 |
|---|---|
| 認証 | **API キー**（Anthropic Console、Tier 2〜4） |
| モデル | 既定（Opus + Sonnet）／実験的に Sonnet 4 で広く拾う構成も |
| 並列度 | `--workers 4 --max-concurrent 16` |
| 予算管理 | `max_budget_usd` をフェーズごとに設定 |
| 想定コスト | ターゲットあたり $30〜100 |

### 6.3 研究：論文再現／クロスモデル比較

| 項目 | 推奨 |
|---|---|
| 認証 | **API キー**（Anthropic + OpenRouter） |
| モデル | Sonnet 4.5（メイン） + Sonnet 4 + DeepSeek R1（比較） |
| 並列度 | `--workers 4 --max-concurrent 32` |
| 想定コスト | RQ1 規模で $400〜600、RQ2 規模で $100〜300 |

### 6.4 大規模実装（Ethereum 全クライアント等）

| 項目 | 推奨 |

| 認証 | **API キー（Tier 4 以上 / Enterprise）** |
| モデル | 既定 |
| 実行環境 | CI（GitHub Actions self-hosted runner、第 10 章参照） |
| 並列度 | `--workers 4 --max-concurrent 64` × 並列ジョブ |

## 7. モデル選定の意思決定フローチャート

```
START
  │
  ├─ Q1: Anthropic 依存を避けたい？
  │    ├─ Yes → DeepSeek R1（OpenRouter 経由）。ただし精度・コストで Sonnet 4.5 に劣ることを了承
  │    └─ No → 続く
  │
  ├─ Q2: 1 ヶ月内のターゲット数は？
  │    ├─ 1〜2 個 → Q3 へ
  │    └─ 3 個以上 → API キー一択
  │
  ├─ Q3: 既に Claude Pro/Max に加入している？
  │    ├─ Yes → そのまま使う（追加課金 0）
  │    └─ No → API キー or サブスク新規
  │
  ├─ Q4: コスト予測性は重要？
  │    ├─ Yes（チーム予算管理等） → サブスク（固定額）
  │    └─ No（多少のブレ許容） → API キー（柔軟）
  │
  ├─ Q5: フェーズ 1〜3（仕様理解）の質を最大化したい？
  │    ├─ Yes → Opus 4.7 を明示指定
  │    └─ No → 既定（Opus 既定 + Sonnet 明示指定）で十分
  │
  └─ Q6: フェーズ 03 で「探索的に新規候補を多く拾いたい」？
       ├─ Yes → Sonnet 4 等の少し古い世代を試す（Property Adherence Effect 利用）
       └─ No → 最新 Sonnet 4.5/4.6
```

## 8. 実用上の注意点とアンチパターン

### 8.1 やってはいけないこと

| アンチパターン | 理由 |
|---|---|
| Haiku でフェーズ 03 を回す | Map → Prove → Stress-Test の構造化推論が破綻し、所見品質が大きく落ちる |
| Sonnet 3.5 でフェーズ 01a〜01e を回す | 仕様→プロパティ生成の品質が低く、後段全体の精度が下がる |
| `max_budget_usd` を初期値のまま放置 | 開発中のプロンプトミスで $200 一気に消費する事故が起きる |
| `--max-concurrent 64` で個人 API キー（Tier 1）を使う | 即座にレートリミットでサーキットブレーカが落ちる |
| サブスクで RQ1 級（10 ターゲット）を回す | 月枠を初日で使い切る |

### 8.2 「動作確認」と「本番監査」を分ける

```bash
# 動作確認用：軽量・低コスト（フェーズ 04 まで動くか検証だけ）
SPEC_URLS="<small spec URL>" \
  uv run python3 scripts/run_phase.py --target 04 \
    --workers 1 --max-concurrent 2

# 本番監査用：通常設定
uv run python3 scripts/run_phase.py --target 04 \
  --workers 4 --max-concurrent 16
```

開発中は「動作確認用」で **小さい仕様＋小さいターゲット** で回すと、$1〜5 程度で 1 周回せます。

### 8.3 PARTIAL ファイルの活用

「フェーズ 01〜02c の結果を使い回す」テクニック：

```bash
# 1. 一度フェーズ 02c まで実行（仕様→プロパティ→コード位置）
uv run python3 scripts/run_phase.py --phase 01a 01b 01e 02c

# 2. PARTIAL ファイル一式をバックアップ
tar -czf snapshot.tar.gz outputs/01*_PARTIAL_*.json outputs/02c_PARTIAL_*.json

# 3. 新しいターゲットでも、仕様コーパスが同じなら 02c まではスキップできる
#    （新しい target_workspace/ を用意してフェーズ 03 から実行）
uv run python3 scripts/run_phase.py --phase 03 04
```

**仕様分析は仕様コーパスに対して 1 回でよい** という性質を最大限活用すれば、N 実装の監査でフェーズ 1〜3 のコストは 1/N に償却されます。

## 9. まとめ

| 観点 | 推奨 |
|---|---|
| **第一選択モデル** | フェーズ 01a〜01e は **Opus 4.x**、フェーズ 02c〜04 は **Sonnet 4.5/4.6** |
| **個人利用** | Claude Pro / Max + 小さく始める |
| **チーム・継続利用** | API キー（Tier 引上げ） |
| **コスト目安** | バグあたり **$1.69**、ターゲット規模で **$30〜100** |
| **代替モデル** | DeepSeek R1 は動くが精度・コストで Sonnet 4.5 に劣る |
| **避けるべき選択** | Haiku でフェーズ 03、Sonnet 3.5 でフェーズ 01〜01e |

SPECA は LLM の能力に **明確に依存** するツールですが、Property Adherence Effect が示すとおり、「最高性能モデル」が必ずしも「最大の発見数」を意味しないという特異性があります。**仕事の性質に応じてモデルを使い分け、PARTIAL ファイルベースの再開を活用してコストを最適化する** のが実用上の鍵です。
