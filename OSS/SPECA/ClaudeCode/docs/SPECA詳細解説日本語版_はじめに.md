## 0.1 SPECA が解こうとしている課題

ソフトウェアのセキュリティ監査ツールの大半は、**コードを起点**にして「既知のバグパターンに似た構造が無いか」をスキャンする方式を採っています。lint 系・SAST、最近のコード駆動 LLM 監査ツールも基本的にこの方向です。

しかし、ブロックチェーンのコンセンサスクライアント、暗号ライブラリ、ネットワークプロトコル実装といった **「仕様駆動システム」** では、しばしば次の現象が起きます：

- バグの正体が「**仕様が要求している不変条件**を実装が満たしていない」ことであり、コードの局所パターンとしては一切怪しくない
- 同じ仕様に対する **複数の独立実装**（例：Ethereum クライアントの 10 種以上）があり、それぞれを別個に監査するのは非効率
- 検出された誤検知（False Positive）に対して「なぜそう判断したのか」が不透明で、改善ターゲットを特定できない

この 3 点は、コード駆動の手法では構造的に解けません。

---

### 例：Sherlock コンテストで 366 人の専門監査者が見落とした暗号バグ

SPECA は、Ethereum の Fusaka アップグレードを対象とした Sherlock 監査コンテスト（10 実装、366 件の専門家提出物）で、**全提出に含まれていなかった暗号学的バグ** を独自に発見しました：

> **c-kzg-4844** — KZG バッチ検証の Fiat-Shamir チャレンジハッシュが、**重複排除前の元のコミットメント配列** を使って計算されている。仕様は「重複排除後の配列で計算」を要求しており、これが満たされないと **選択的偽造** が可能になる。

このバグは、コード上は「ハッシュ関数を呼んでいる正常なロジック」に見え、コードパターンマッチでは検出不能です。一方、仕様には明確に「dedup されたコミットメントから challenge を作る」と書かれており、**仕様→プロパティ** という方向で監査すれば自然に検出できます。SPECA はまさにこの方向で動きます。

## 0.2 既存ツール（コード駆動の監査ツール）との立ち位置

| 観点 | コード駆動監査（既存 LLM ツール含む） | SPECA（仕様アンカー型） |
|---|---|---|
| **検出の起点** | リポジトリ内の「バグらしいパターン」を探索 | 仕様から導出された **明示的な型付きプロパティ** の違反を探索 |
| **クロス実装比較** | 各コードベースを独立に解析 | 単一のプロパティ語彙を **N 個の実装に均一適用** |
| **誤検知の説明性** | 不透明（「モデルがバグだと思った」） | 3 つの根本原因（信頼境界誤解／コード読解誤り／仕様解釈誤り）に分解可能、各原因はパイプラインの特定フェーズに紐付く |
| **検出の証拠（プロブナンス）** | 必ずしも辿れない | `所見 → プロパティ → サブグラフ → 仕様セクション → INV-* ラベル` の連鎖が常に残る |

この 4 列目（プロブナンス）が SPECA の隠れた強みです。**監査結果が常に「仕様のどの記述から派生したのか」を辿れる** ため、レビュー時に「これは本当に仕様要件か？」を確認するコストが大幅に下がります。

## 0.3 本資料の構成・用語

### 構成

本資料は 12 のファイルに分かれます：

- **概略と目次.md**（本資料の入り口）
- **第 0 章 はじめに**（このファイル）
- **第 1 章 全体像とコンセプト** — 思想・パイプライン全体像
- **第 2〜8 章** — 実装サイドの詳細（リポジトリ構成、各フェーズ、オーケストレータ、CLI、サーバ等）
- **第 9 章 実績とベンチマーク** — マネージャー必読パート（数字・誤検知タクソノミー・コスト）
- **第 10〜11 章** — 運用・適用ガイド

各章は独立して読めるように書かれています。**マネージャー層は 第 0／1／9／11 章** を、**エンジニア層は 第 2〜8／10 章** を中心に読むのが効率的です。

### 主要用語

本資料を読み進めるために、最低限知っておきたい用語を整理しておきます。

| 用語 | 意味 |
|---|---|
| **仕様（Specification）** | EIP、RFC、コンセンサス仕様などの自然言語文書。SPECA はこれを起点とする |
| **プログラムグラフ（Program Graph）** | Nielson & Nielson の形式定義に基づく、仕様中の手続きの状態遷移グラフ。SPECA では拡張版 Mermaid 状態遷移図で表現 |
| **サブグラフ（Subgraph）** | 仕様中の 1 つの手続き（関数）に対応する 1 つのプログラムグラフ |
| **プロパティ（Property）** | 仕様から導出された型付きセキュリティ要件。`invariant` ／ `pre-condition` ／ `post-condition` ／ `assumption` の 4 種 |
| **証明試行（Proof Attempt）** | 「プロパティが成り立つことを証明しようと試み、証明が破綻する箇所をバグとして報告する」推論スタイル。SPECA の中核手法 |
| **3 ゲートフィルタ** | フェーズ 04 が実施する誤検知フィルタ。Dead Code／Trust Boundary／Scope Check の 3 段階 |
| **PARTIAL ファイル** | 各バッチの結果を即座に保存する JSON。再開と部分進捗保存の鍵 |
| **MCP（Model Context Protocol）** | Claude Code が外部ツールを呼び出すためのプロトコル。SPECA は `tree_sitter` ／ `filesystem` ／ `fetch` の 3 種を使用 |
| **スキル（Skill）** | Claude Code の機能で、`.claude/skills/<name>/SKILL.md` に定義される再利用可能な手続き |

### フェーズ ID と論文上の章番号の対応

SPECA のフェーズには 2 系統の番号体系があり、これが混乱を招きやすいので最初に整理します：

| 論文の Phase 番号 | リポジトリの Phase ID | 名称（日本語） |
|---|---|---|
| Phase 1 | `01a` | 仕様文書の発見 |
| Phase 2 | `01b` | サブグラフ抽出 |
| Phase 3 | `01e` | プロパティ生成 |
| Phase 4 | `02c` | コード位置の事前解決 |
| Phase 5 | `03` | 監査マップ生成 |
| Phase 6 | `04` | 監査レビュー |

**本資料では、原則としてリポジトリの Phase ID（`01a`〜`04`）を使用します。** 論文との対応が必要な箇所は併記します。

## 0.4 SPECA を「触ってみたい」読者へ

SPECA はローカルで動作する OSS です。詳細は第 10 章で扱いますが、最短コースだけ示しておくと：

```bash
git clone https://github.com/NyxFoundation/speca.git
cd speca
npm install -g @anthropic-ai/claude-code   # ワーカランタイム
uv sync                                     # Python 依存
bash scripts/setup_mcp.sh                   # MCP サーバ登録

# シングルフェーズ実行
SPEC_URLS="https://github.com/ethereum/EIPs/blob/master/EIPS/eip-7594.md" \
  uv run python3 scripts/run_phase.py --phase 01a
```

エンドツーエンド実行には `outputs/BUG_BOUNTY_SCOPE.json`（信頼モデル＋重要度ルーブリック）と `outputs/TARGET_INFO.json`（対象リポジトリと固定コミット）の 2 ファイルが必要です。フォーマットは第 10 章に詳述します。
