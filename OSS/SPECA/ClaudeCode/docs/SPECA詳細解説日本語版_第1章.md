# SPECA 詳細解説（日本語版） — 第 1 章 全体像とコンセプト

> 関連：[概略と目次](概略と目次.md) ／ [前：第 0 章 はじめに](SPECA詳細解説日本語版_はじめに.md) ／ [次：第 2 章 リポジトリ構成](SPECA詳細解説日本語版_第2章.md)

本章では、SPECA がどのような **思想** で組み立てられているか、なぜそうなっているかを解説します。実装の詳細（ディレクトリ構成・コード）には踏み込まず、まず「設計の根」を共有することを目的とします。

---

## 1.1 仕様アンカー型監査という発想

### 1.1.1 既存ツールが抜け落とす領域

LLM ベースの監査ツールは、ここ数年で急速に発展しました。RepoAudit、CodeGuru、Infer、各種 SAST ツール — これらはいずれも **「コードを起点に、バグらしいパターンを探す」** という方向で設計されています。

この方向は、典型的なバグ（NPD、メモリリーク、UAF、SQL インジェクション、…）には強力です。しかし、次のようなクラスのバグには構造的に弱いことが知られています：

- **仕様の数式・不変条件を実装が満たしていない** タイプのバグ
- **複数のコードパス・複数のモジュールにまたがる仕様要件**（局所パターンとして見えない）
- **データ依存ではなく仕様依存** のバグ（例：「この計算は dedup 後の配列で行え」）

このクラスのバグは、Ethereum コンセンサスクライアント、暗号ライブラリ、コンセンサスプロトコル実装などの「**仕様駆動システム**」で頻繁に登場します。

### 1.1.2 SPECA の方向転換

SPECA は分析の方向を反転させました：

```
従来:    [実装コード] ─► バグらしいパターンを探す
SPECA:  [仕様]  ─► 型付きプロパティ ─► 各プロパティの違反を実装で探す
```

この方向反転が生み出すものは、単なる「精度向上」ではありません。**「コードを見ても気づきようがない仕様起因のバグ」を検出可能にする** という、定性的に新しい能力です。

## 1.2 2 ステージ 6 フェーズのパイプライン

SPECA は全体で 6 つのフェーズから成り、これを 2 つのステージに分けて運用します。

### 1.2.1 全体像

```
==================  ステージ 1: 知識の構造化  ==================
                  （仕様 1 つに対して 1 回実行）

  仕様文書         サブグラフ          プロパティ
  発見    ─────►  抽出      ─────►   生成
  (01a)            (01b)              (01e)

==================  ステージ 2: 体系的監査  ==================
                  （実装ごとに繰り返し実行）

  コード位置       監査マップ          監査
  事前解決  ───►  生成（証明試行）─► レビュー
  (02c)            (03)               (04)
```

| 論文 Phase | リポジトリ Phase ID | フェーズ名 | 主な役割 |
|---|---|---|---|
| 1 | `01a` | 仕様文書の発見 | 仕様サイトをクロール、仕様カタログを作成 |
| 2 | `01b` | サブグラフ抽出 | 各仕様文書を Nielson & Nielson 風のプログラムグラフへ分解 |
| 3 | `01e` | プロパティ生成 | STRIDE+CWE Top 25 に基づき、サブグラフから型付きセキュリティプロパティを導出 |
| 4 | `02c` | コード位置の事前解決 | 各プロパティに対応する実装上のコード位置（ファイル／関数）を Tree-sitter で先に決める |
| 5 | `03` | 監査マップ生成 | プロパティごとに **証明試行** を実行（Map → Prove → Stress-Test の 3 サブフェーズ） |
| 6 | `04` | 監査レビュー | 3 ゲートでの誤検知フィルタ＋重要度キャリブレーション |

### 1.2.2 ステージ分割が生む実利

ステージ 1（仕様の知識構造化）は **仕様 1 つに対して 1 回だけ** 動かせばよく、ステージ 2 はそれを使い回して **N 個の実装** を順に監査できます。Ethereum のように 10 種類のクライアント実装が同じ仕様を共有する世界では、これが大きな効率を生みます：

- 仕様分析コストは N 実装で 1/N に償却される
- すべての実装が **同じプロパティ語彙** で監査される ──「クライアント A だけ厳しくチェックされた／甘くチェックされた」というブレが構造的に発生しない
- 新しい実装が登場した時、ステージ 2 だけを動かせば監査できる

これは「**統制されたクロス実装比較**」と呼ばれ、論文の 3 つの能力のうち 2 番目の柱です。

## 1.3 「証明試行」を採る理由 — 88% FP の失敗からの学び

### 1.3.1 最初は普通の「バグを探せ」プロンプトだった

SPECA の初期プロトタイプは、典型的な敵対的フレーミング — **「このコードのバグを見つけろ」** というプロンプト — を採用していました。

結果は悲惨でした：

> **誤検知率 88%。**
> 構造化された主張（disprove 対象）が無い場合、モデルは弱い根拠で投機的な所見を大量に出力する。

これは、SPECA に限らず LLM ベース監査ツール全般が抱える問題です。「自由にバグを探せ」と言われた LLM は、しばしば **「もしかしたら問題かもしれない」レベルの推測** を「バグ」として報告してしまいます。

### 1.3.2 解決策：プロパティを「証明しようと試みる」

SPECA は、プロンプトの構造をひっくり返しました：

```
従来:   「このコードのバグを探して、見つけたら報告して」
SPECA:  「このプロパティが成り立つことを証明しろ。証明が破綻する箇所を報告しろ」
```

この変更は「言葉のあやい」ではなく、推論の構造そのものを変えます：

| 観点 | 「バグを探せ」 | 「プロパティを証明せよ」 |
|---|---|---|
| 出発点 | 漠然とした不安 | 明確な検証可能主張 |
| 終了条件 | 主観的（「もう見つからなさそう」） | 客観的（証明が完了するか、ギャップが見つかる） |
| 報告される所見 | 主観的「バグらしさ」 | 「この主張のここが証明できなかった」という具体的ギャップ |
| 誤検知発生率 | 88% （プロトタイプ実測） | 30% 前後（フェーズ 04 後） |

> **重要な注意：** ここでいう「証明」は **形式手法（Coq・Isabelle 等）による厳密な定理証明ではなく**、LLM が **構造化された推論ステップで証拠を構築する作業** のことを指します。論文では `LLM-driven evidence construction with structured reasoning steps` と表現されています。
>
> 構造化されているからこそ、検出も失敗も **後から分析可能** になる、というのが本質です。

### 1.3.3 3 サブフェーズ — Map / Prove / Stress-Test

フェーズ 03（監査マップ生成）は、1 つのプロパティに対して 3 段階の処理を行います：

```
[Map 段階]
  プロパティの assertion を検証可能なサブクレームに分解し、
  対応するコードを完全に読んで（呼出元・呼出先含めて）紐付ける

[Prove 段階]
  各サブクレームに対し、入力カバレッジ・パスカバレッジ・並行性安全・
  時間的妥当性・実装パターン上の義務を検証する。証明できない箇所は
  「ギャップ＝所見」として記録される

[Stress-Test 段階]
  証明が成功したなら：すべての仮定を再検査して反例を作ろうとする
  証明が失敗したなら：具体的な攻撃パスを構築できるか挑戦する
  攻撃パスを作れない所見は potential-vulnerability に格下げ
```

この 3 段階構造によって、**「成り立つ」「成り立たない」「成り立たないが攻撃には届かない」** の 3 値判定が自然に出てきます。

## 1.4 SPECA が獲得した 3 つの能力

論文では、SPECA がコード駆動監査では得られない 3 つの能力を獲得したと主張しています。本資料はそのうち最初の 1 つを 1.1 節で、2 つ目を 1.2.2 節で扱いました。3 つ目は誤検知の説明性です。

### 能力 1: 仕様依存検出（Spec-Dependent Detection）

仕様にしか書かれていない不変条件の違反を検出できます。Sherlock コンテストで 366 件の専門家提出物に含まれていなかった **c-kzg-4844 の Fiat-Shamir チャレンジハッシュバグ** はこのカテゴリの代表例です。コードレベルではどこにも怪しさが無く、仕様の「dedup 後の配列で hash を計算せよ」という記述に照らして初めて違反が見えます。

### 能力 2: 統制されたクロス実装比較（Controlled Cross-Implementation Comparison）

同じプロパティ集合で N 実装を均一に監査するため、「実装間の挙動差」を構造的に比較できます。RQ1 では 10 個の Ethereum クライアントに対してこれが実証されました。

### 能力 3: 誤検知の根本原因分解（Decomposable False Positives）

これは SPECA 固有の重要な性質です。RQ1 で 16 件の FP を深く分析した結果、**誤検知のすべてが 3 つの根本原因のいずれかに分類できる** ことが判明しました：

| 根本原因 | 発生フェーズ | N | % |
|---|---|---|---|
| 信頼境界の誤解 | フェーズ 3（プロパティ生成） | 8 | 50.0% |
| コード読解の誤り | フェーズ 5（監査） | 6 | 37.5% |
| 仕様解釈の誤り | フェーズ 3 | 2 | 12.5% |

この分解は実用的に重要です。**「FP を減らしたい」が「どう減らすか分からない」では改善できないが、「信頼境界の誤解を減らせばよい」ならば改善ターゲットが明確になります**。実際、論文ではこの 3 原因が次の 3 つの実装可能な改善目標に直接マップすると主張しています：

- **信頼境界の明示的設定** — `BUG_BOUNTY_SCOPE.json` の `trust_assumptions` の充実（既に実装済み）
- **コード読解コンテキストの強化** — フェーズ 02c の Tree-sitter 事前解決と、フェーズ 03 の完全関数読込み
- **仕様セクションの再読込みの強制** — プロパティ分類前の参照仕様の再確認

## 1.5 もう一つの柱：プロブナンス（来歴）

SPECA のもう一つの重要な性質は、**すべての所見が「仕様のどこから来たか」を辿れる** ことです。

```
所見 (Finding-04)
   ↑ 由来
プロパティ (PROP-6a4369e9-inv-042: "data_column_sidecars_by_root must reject ...")
   ↑ 由来
サブグラフ (SG-024: "Polynomial Commitments Sampling")
   ↑ 由来
仕様セクション (Fulu Polynomial Commitments Sampling spec § 3.2)
   ↑ 由来
INV-* ラベル (INV-001, INV-002, ...)
```

この連鎖が常に保存されているため、レビュア／監査クライアント／開発者が **「これは本当に仕様要件か？」を確認するのに、フェーズの最後ではなく最初に戻れる** という利点があります。

## 1.6 「証明試行」が当たり前ではない理由

最後に、なぜこの設計が他の LLM 監査ツールに広く採用されていないか、という疑問に触れておきます。

「証明試行」を機能させるには、**前段で「何を証明するか」を明示的に決める仕組み** が必要です。SPECA の場合、それがフェーズ 01a〜01e（仕様→サブグラフ→プロパティ）です。**プロパティ生成のないコード駆動ツールは、そもそも「証明試行」プロンプトを使えない** のです。

逆に言うと、SPECA の本当の発明は「証明試行プロンプト」ではなく、**「コード駆動から仕様駆動へ転換することで、プロパティを定義できる土台を作った」** ことです。証明試行はその副産物として自然に出てきます。

---

## 1.7 章のまとめ

- SPECA はコード駆動ではなく、**仕様→プロパティ→実装** の方向で監査する
- パイプラインは 6 フェーズ・2 ステージ。ステージ 1 は仕様 1 つに対して 1 回、ステージ 2 は実装ごと
- 中核手法は **「証明試行」** — 88% FP を出した「バグを探せ」プロンプトの反省から設計された
- 結果として SPECA は 3 つの能力（仕様依存検出／クロス実装比較／FP の根本原因分解）を獲得している
- すべての所見は **仕様セクションまでの来歴** を持つ

次章では、これらの思想がコードベース上でどのように構成されているかを見ていきます。
