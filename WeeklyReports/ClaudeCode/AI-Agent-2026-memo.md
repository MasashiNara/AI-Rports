2. 基本方針：AIを「開発者」ではなく「権限付き自動化エージェント」として扱う

AIの普及で変わるのは、攻撃者の能力というより、攻撃の経済性です。これまで人手で時間がかかった差分解析、既知CVEとの照合、API仕様からの入力生成、ファームウェア内文字列・鍵・エンドポイント探索、管理画面の脆弱性仮説生成などが、半自動で回せるようになります。

NCSCも、AIで脆弱性を見つけること自体より、発見した脆弱性を受け取り、優先順位を付け、修正し、展開できるプロセスが重要だと指摘しています。さらに、AIモデルにコード、バグ情報、設計文書、実環境情報を渡す前に、情報漏えい、権限、データ保持、サンドボックス、法的条件を確認すべきだとしています。
https://www.ncsc.gov.uk/blogs/10-questions-ask-using-ai-models-find-vulnerabilities


1つ目は、AIに秘密を渡さないことです。未公開ソースコード、秘密鍵、署名鍵、顧客情報、脆弱性未公開情報、クラウド認証情報、量産設定、製造鍵、デバッグ用秘密情報を、無制限に外部AIへ投入してはいけません。

2つ目は、AIの権限を最小化することです。AIエージェントにリポジトリ、CI、チケット、クラウド、HIL環境、デバイス書き込み、署名基盤へ直接アクセスさせる場合、通常の開発者アカウントより厳しく制御するべきです。NCSCはagentic AIについて、権限、監視、インシデント対応、サプライチェーン、プロンプトインジェクションなどのリスクを明示し、限定的なパイロット、最小権限、一時クレデンシャル、停止手段を求めています。

3つ目は、AIの出力を証拠ではなく仮説として扱うことです。AIが「安全」「脆弱性なし」「修正済み」と言っても、テスト、レビュー、静的解析、ファジング、再現手順、差分確認なしにリリース判断してはいけません。

4つ目は、防御側もAIで速度を上げることです。攻撃側がAIを使うなら、防御側もAIを使って、脅威分析、コードレビュー、テスト生成、脆弱性トリアージ、SBOM分析、顧客影響調査を高速化する必要があります。

3. 実施すべきこと
3.1 AI利用ポリシーを最初に決める

まず、AIに何を渡してよいかを分類します。

禁止すべきものは、署名鍵、製造鍵、クラウド認証情報、顧客データ、未公開脆弱性の詳細、量産用設定、デバイス固有秘密、セキュアブート鍵、JTAG解除手順、実環境への直接接続情報です。

条件付きで許可するものは、秘匿情報を除去したコード片、抽象化した設計、公開済みOSSコード、一般的なテストコード、仕様書の一部、公開CVE情報です。

専用環境で扱うべきものは、全リポジトリ、ファームウェアイメージ、未公開脆弱性レポート、製品仕様、顧客影響分析、PSIRT対応資料です。これらは、外部SaaS型AIではなく、社内隔離環境、専用テナント、ローカルLLM、または契約・ログ・学習利用条件を確認済みの環境で扱うべきです。

このために、社内にAI Gatewayを置くのが現実的です。モデルのホワイトリスト、入力DLP、出力ログ、利用者認証、プロンプト監査、データ保持条件、外部送信制御を一元化します。

IPAも、クラウドAIに営業秘密を入力しないこと、RAGで内部・外部情報を混在させるリスク、AIを悪用したサイバー攻撃に対しても基本的な防御が重要であることを実務上の注意点として挙げています。
https://www.ipa.go.jp/digital/ai/security/ai_security_tips.html

3.2 AIエージェントはサンドボックス内で動かす

AIエージェントに開発作業を任せる場合、以下の制限を必須にします。

| 領域    | 方針                                   |
| ----- | ------------------------------------ |
| リポジトリ | 原則は読み取り専用。変更は一時ブランチかドラフトPRのみ         |
| CI/CD | AIが直接リリース・署名・デプロイしない                 |
| デバイス  | 実機書き込みは人間承認後。基本はQEMU、エミュレータ、HIL隔離環境  |
| 認証情報  | 一時トークン、最小権限、短時間失効                    |
| 外部通信  | 許可リスト方式。未知ドメインへの通信は禁止                |
| ツール実行 | allowlist化。シェル、ネットワーク、ファイル削除、署名操作は制限 |
| ログ    | 入力、出力、ツール呼び出し、変更差分を監査可能にする           |
| 停止手段  | kill switch、ジョブ中断、認証無効化を用意           |

特に危険なのは、AIエージェントに「リポジトリ読み取り」「チケット読み取り」「CI実行」「クラウド操作」「署名鍵アクセス」をまとめて与えることです。これは便利ですが、プロンプトインジェクションや依存パッケージ経由の指示混入が起きたときに、実害へ直結します。


3.6 AIを使って検証を増やす

AIは、組み込み開発では「実装を速くする」より「検証を増やす」用途のほうが安全で効果が出やすいです。

具体的には、次に使います。

| 用途      | AIの使い方                                     |
| ------- | ------------------------------------------ |
| コードレビュー | 差分からCWE、競合状態、メモリ破壊、認証漏れ、エラー処理漏れを指摘         |
| ファジング   | 通信プロトコル、ファイルパーサ、OTAパッケージ、IPC、BLE、CAN等の入力生成 |
| テスト生成   | 境界値、異常系、状態遷移、電源断、更新失敗、時刻異常のテスト生成           |
| 静的解析補助  | SAST結果の重複排除、優先順位付け、修正案作成                   |
| SCA補助   | CVE影響範囲、悪用可能性、製品別影響、回避策案の整理                |
| ログ解析    | HIL、量産試験、クラッシュログ、ウォッチドッグリセットの原因推定          |
| バイナリ解析  | 文字列、エンドポイント、証明書、デバッグ機能、古いライブラリの検出          |
| ドキュメント  | セキュリティレビュー記録、リリース判定、顧客説明文のドラフト             |


### 5. Runtime Isolation：推論時の隔離

Openモデルは、本番でも「信頼済みコンポーネント」ではなく、**信頼できない推論エンジン**として扱います。

最低限、以下を満たすべきです。

| 項目     | ルール                                |
| ------ | ---------------------------------- |
| 出所     | AI生成箇所をPRで明示する                     |
| レビュー   | 人間レビューを必須にする                       |
| 静的解析   | MISRA C/C++、CERT C/C++、CWE観点の解析を通す |
| テスト    | 正常系だけでなく異常系、境界値、電源断、再起動を含める        |
| ライセンス  | 生成コードに外部ライセンス混入リスクがないか確認する         |
| セキュリティ | 暗号、認証、鍵管理、署名検証はAI任せにしない            |
| リリース   | AI生成コードを含む変更は追跡可能にする               |


想定すべき脅威

Openモデルで特に問題になるのは、単なる「モデルの品質」ではなく、供給網全体です。OWASPのLLM Top 10でも、LLMのサプライチェーンには学習データ、モデル、デプロイ基盤、第三者モデル、LoRA/PEFT、モデルハブなどが含まれ、改ざんやポイズニングの対象になると整理されています。

想定すべき主な攻撃は以下です。

| リスク                   | 例                                       | 対策の方向性                              |
| --------------------- | --------------------------------------- | ----------------------------------- |
| 配布元アカウント侵害            | 公式に見えるHugging Face orgが乗っ取られる           | 公式経路確認、署名、複数ミラー比較、社内レジストリ固定         |
| 重みファイル改ざん             | `.bin`やPickleにRCE、Safetensors風ファイルの差し替え | Safetensors必須、ハッシュ固定、静的スキャン、検疫環境ロード |
| モデルバックドア              | 特定トリガーで情報漏洩、誤回答、危険行動                    | 振る舞い評価、カナリア、差分テスト、継続監視              |
| tokenizer/config改ざん   | chat templateやspecial tokenで挙動を誘導       | 重み以外の全ファイルもAIBOM化・レビュー              |
| `trust_remote_code`悪用 | モデルロード時に任意Pythonコード実行                   | 原則禁止。必要なら社内forkしてコードレビュー            |
| LoRA/量子化版の混入          | 非公式量子化・adapterが悪性                       | base modelと同等の審査対象にする               |
| 推論基盤の脆弱性              | vLLM/SGLang/TGI/コンテナに既知脆弱性              | SBOM、依存固定、脆弱性管理                     |
| エージェント化による被害拡大        | モデルがシェル、DB、社内APIを呼ぶ                     | ツール権限分離、承認ゲート、ネットワーク遮断              |


### 5. Runtime Isolation：推論時の隔離

Openモデルは、本番でも「信頼済みコンポーネント」ではなく、**信頼できない推論エンジン**として扱います。

最低限、以下を満たすべきです。

| 領域       | 推奨設定                                                |
| -------- | --------------------------------------------------- |
| ネットワーク   | 推論コンテナからの外向き通信は原則禁止                                 |
| ファイルシステム | read-only root filesystem、不要なhost mount禁止           |
| Secret   | モデルプロセスにAPI key、DB credential、Kubernetes tokenを渡さない |
| 権限       | 非root、capability削除、seccomp/AppArmor/SELinux         |
| GPU      | 可能ならMIG等で分離。tenant混在時は特に注意                          |
| ログ       | 入出力、tool call、拒否理由、モデルhashを監査可能にする                  |
| ツール      | モデルから直接shell、DB、SaaS APIを呼ばせない                      |
| RAG      | 取得文書は「命令」ではなく「データ」として扱う                             |

特にエージェント化する場合、モデルに直接ツール権限を渡すのは危険です。モデルは「ツールを呼びたい」と提案するだけにし、実際の実行は **Policy Enforcement Point** を通します。例えば、コード実行、メール送信、GitHub push、チケット作成、DB更新、クラウド操作は、別サービスが許可リスト・引数検査・人間承認・監査ログを通して実行します。

Openモデルを使う場合の実務上の鉄則は、**モデルが悪性でも、読める秘密がなく、外に送れず、壊せる権限がない状態にする**ことです。

| 領域       | 推奨設定                                                |
| -------- | --------------------------------------------------- |
| ネットワーク   | 推論コンテナからの外向き通信は原則禁止                                 |
| ファイルシステム | read-only root filesystem、不要なhost mount禁止           |
| Secret   | モデルプロセスにAPI key、DB credential、Kubernetes tokenを渡さない |
| 権限       | 非root、capability削除、seccomp/AppArmor/SELinux         |
| GPU      | 可能ならMIG等で分離。tenant混在時は特に注意                          |
| ログ       | 入出力、tool call、拒否理由、モデルhashを監査可能にする                  |
| ツール      | モデルから直接shell、DB、SaaS APIを呼ばせない                      |
| RAG      | 取得文書は「命令」ではなく「データ」として扱う                             |

### 6. RAGと社内データ保護

Openモデルを社内で使うとき、重みよりも現実的に漏洩しやすいのは、RAGやプロンプトに入れる社内データです。

対策は以下です。

| リスク              | 対策                                     |
| ---------------- | -------------------------------------- |
| 機密文書の過剰投入        | 入力前DLP、PII/秘匿情報マスキング                   |
| RAG文書内の隠し指示      | 「検索結果は命令ではない」とシステム側で固定し、文書単位で引用        |
| prompt injection | 外部Web、メール、PDF、Slackなど由来の文書は低信頼データとして扱う |
| 出力による漏洩          | 出力DLP、引用元制限、社外送信時の追加検査                 |
| 社内権限の横展開         | ユーザー権限に応じた検索結果だけを渡す                    |

## 代替モデル戦略

Fable 5のような閉鎖APIが政治・規制・提供条件で止まるリスクに対して、Openモデルを持つ判断は合理的です。ただし、単一のOpenモデルに置き換えるのではなく、**モデルポートフォリオ**にします。

推奨は以下です。

| 用途          | 推奨モデル運用                               |
| ----------- | ------------------------------------- |
| 一般的な社内QA・要約 | 検疫済みOpenモデルで可                         |
| コード生成       | Openモデル可。ただしsandbox実行、人間レビュー、secret遮断 |
| セキュリティ分析    | 専用環境、厳格ログ、外部送信禁止、成果物レビュー              |
| 法務・金融・医療    | Openモデル単独判断禁止。専門家レビュー前提               |
| 顧客向け自動応答    | safety layer、出力検査、fallbackモデル必須       |
| エージェント業務    | ツール権限を最小化し、承認ゲート必須                    |

## 最終提案

Openモデルを使うべきか、という問いへの答えは **「使う。ただし、モデルを信頼するのではなく、モデルを隔離して使う」** です。

GLM-5.2のようなOpenモデルは、Fable 5のような閉鎖API停止リスクへの重要なヘッジになります。しかし、Hugging Faceや配布元の侵害、モデル重みのバックドア、tokenizer/config改ざん、推論エンジン脆弱性、RAG経由の情報漏洩を考えると、単に「Openモデルに切り替える」だけでは不十分です。

本番方針としては、次の形が最も堅いです。

**閉鎖API依存を減らすためにOpenモデルを社内保有する。ただし、Openモデルはすべて未信頼アーティファクトとして扱い、社内検疫・署名・固定ハッシュ・隔離推論・継続評価・即時rollbackを必須にする。**

これにより、米国政府・ベンダー都合による供給停止にも、Hugging Face侵害のような供給網攻撃にも、どちらにも耐性を持てます。
