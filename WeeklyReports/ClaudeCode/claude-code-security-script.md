# 全体の話し方

この資料では、細かい機能説明よりも、社員に次の3点を覚えてもらうことをゴールにすると伝わりやすいです。

> **レビューする、権限を絞る、隔離する。**
> Claude Code は便利だが、開発作業を実行できるエージェントなので、通常の開発セキュリティと同じように扱う。

各ページでは、スライドの文字を読み上げすぎず、**「つまり何をすればよいか」**を一言で補足する流れにするとよいです。

---

# 1枚目: 表紙

## Claude Code を安全に使うための 3 つの約束

**想定時間: 15〜20秒**

### 台本

> 今日は、Claude Code を安全に使うための基本ルールを簡単に紹介します。
>
> Claude Code は、単なるチャットではなく、コードを読んだり、ファイルを書き換えたり、コマンドを実行したりできる開発エージェントです。
>
> そのため、便利さを活かすには、使い方のルールも必要です。今日覚えていただきたいのは、ここにある3つです。
>
> **レビューする、権限を絞る、隔離する。**
>
> この3つを軸に、5分ほどで全体像を説明し、後半でDockerやDev Containerを使った隔離の考え方も紹介します。

### 話すポイント

ここでは詳しく説明しすぎず、**「Claude Code は強い権限を持ち得るツール」**という前提だけ置きます。

# 2枚目: 01 なぜセキュリティが必要か

## Claude Code は「チャット」ではなく開発作業を実行するエージェント

**想定時間: 45秒**

### 台本

> まず、なぜセキュリティが必要なのかです。
>
> Claude Code は、普通のチャットAIとは少し違います。コードベースを読み、ファイルを編集し、テストやビルドなどのコマンドも実行できます。
>
> これは開発効率を大きく上げる一方で、リスクもあります。たとえば、意図しないコード変更、秘密情報の読み取り、危険なBashコマンドの実行、外部通信による情報流出などです。
>
> つまり、Claude Code を使うということは、AIに開発作業の一部を任せるということです。任せる範囲が広がるほど、アクセスできるファイルや実行できる操作を絞る必要があります。
>
> 今日の話は、Claude Codeを怖がるためではなく、安全に便利に使うための基本ルールです。

### 話すポイント

このページの一言メッセージはこれです。

> **AIに任せる作業が増えるほど、権限管理が重要になる。**

---

# 3枚目: 02 守るべき 3 つのレイヤー

## 安全対策は 1 個ではなく、3 層で考える

**想定時間: 60秒**

### 台本

> Claude Code の安全対策は、1つの機能だけで完結するものではありません。
>
> ここでは3つのレイヤーで考えます。
>
> 1つ目は、**レビュー層**です。security-guidance プラグイン、PRレビュー、CIや静的解析ツールなどで、問題を見つけます。
>
> 2つ目は、**権限制御層**です。Claude Code が読めるファイル、実行できるコマンド、許可する操作を絞ります。たとえば `/permissions` や `permissions.deny`、secretファイルの読み取り禁止などです。
>
> 3つ目は、**隔離層**です。built-in sandbox、Dev Container、Docker、VMなどを使って、Claude Code の実行環境をホスト環境から切り離します。
>
> 重要なのは、それぞれ役割が違うことです。レビューは問題を見つける仕組みです。一方で、権限やsandboxは、そもそも危険な操作ができないようにする、または被害範囲を小さくする仕組みです。

### 話すポイント

このページでは、次の言い回しが伝わりやすいです。

> **レビューは「見つける仕組み」。権限とsandboxは「行動を制限する仕組み」。**

---

# 4枚目: 03 security-guidance プラグイン

## Claude Code が書いたコードを、その場でレビュー

**想定時間: 75秒**

### 台本

> 1つ目のレイヤー、レビュー層の代表が、公式の `security-guidance` プラグインです。
>
> これは、Claude Code が書いたコードを、そのセッション内でセキュリティ観点からレビューするためのプラグインです。
>
> チェックのタイミングは主に3つあります。
>
> まず、ファイル編集時です。ここでは危険なパターンを高速にチェックします。たとえば、危険なDOM操作や、危険なコマンド実行、デシリアライズなどのパターンです。
>
> 次に、ターン終了時です。Claude が行った変更のdiffを、別のClaudeがレビューします。つまり、同じ会話の中で、変更内容をもう一度セキュリティ目線で見直すイメージです。
>
> さらに、Claude が commit や push を行う場合には、周辺コードも含めてより深く確認します。
>
> ただし、ここが大事です。これは自動ブロッカーではありません。見逃しもあります。PRレビューやCI、静的解析の代わりにはなりません。
>
> あくまで、早めに気づくための安全ネットです。

### コマンド部分を指しながら補足

> 導入は、公式マーケットプレイスを追加して、`security-guidance` プラグインをインストールし、プラグインをリロードする流れです。
>
> ただし、今日はコマンドを覚えることよりも、**security-guidance はレビュー層であり、最後の砦ではない**という位置づけを覚えてください。

### 話すポイント

このページの締めは短く言うとよいです。

> **security-guidance は早期発見のための安全ネット。過信しない。**

---

# 5枚目: 04 利用者が守るルール

## 開発者が守る 5 つのルール

**想定時間: 75秒**

### 台本

> 次に、実際に使う社員の皆さんに守ってほしいルールです。
>
> 1つ目、Claude の提案をそのまま approve しないこと。
> AIが提案したから正しい、ではなく、通常の開発者が出した変更と同じように確認してください。
>
> 2つ目、`git diff` を見てから承認すること。
> 特に認証、認可、課金、個人情報、CI/CD、インフラ設定に関わる変更は、必ず差分を確認してください。
>
> 3つ目、`.env`、秘密鍵、認証情報を読ませないこと。
> Claude Code に限らず、AIツールに秘密情報を渡さないのが基本です。
>
> 4つ目、よく分からないBashコマンドは許可しないこと。
> コマンドの意味が分からない場合は、実行前に止めてください。特に `curl`、`wget`、削除コマンド、権限変更、外部送信を伴うものは注意が必要です。
>
> 5つ目、機密リポジトリでは sandbox や container を使うこと。
> ローカルPCの通常環境で何でも実行させるのではなく、隔離された環境で動かすことを推奨します。
>
> 要するに、AIだから特別扱いするのではなく、新しい開発者に権限を渡すときと同じ慎重さで扱う、ということです。

### 話すポイント

このページでは、社員に行動として残るように、次の一文を強調します。

> **承認ボタンを押す前に、diff とコマンドを見る。**

# 6枚目: 05 社内標準案

## 初期設定の推奨案

**想定時間: 45秒**

### 台本

> ここでは、社内での初期設定案を示しています。
>
> まず、`security-guidance` プラグインを有効化します。
>
> 次に、`.env` や `secrets/**` のような秘密情報を含む可能性があるファイルは、読み取り禁止にします。
>
> また、不明な `curl` や `wget` は原則確認する運用にします。外部からスクリプトを取得して実行するような操作は、特に慎重に扱うべきです。
>
> 機密リポジトリでは、Dev Container や Docker を使って、Claude Code の実行環境を分離します。
>
> そして最後に、PRレビューやCIのチェックは継続します。ここが非常に重要です。
>
> `security-guidance` を入れたから安全、ではありません。これはセッション中の早期検知の仕組みであり、後段のレビューやCIを置き換えるものではありません。

### コード例の説明

> 右側の設定例では、プラグインを有効化し、`.env` や `secrets/**` の読み取りを禁止し、危険になりやすいコマンドをdenyする形を示しています。
>
> こうした設定を組織として標準化していくことで、利用者ごとの判断ミスを減らせます。

### 話すポイント

このページの一言メッセージは次です。

> **プラグイン、deny設定、container、PR/CIをセットで考える。**

# 7枚目: 06 Docker / Dev Container の意味

## Docker 化の目的: Claude Code をホストから切り離す

**想定時間: 60秒**

### 台本

> ここからは、プラス2分の補足として、DockerやDev Containerでsandboxを実現する考え方を紹介します。
>
> Docker化の目的は、Claude Code をホストPCから切り離すことです。
>
> built-in Bash sandbox は、主にBashコマンドとその子プロセスの影響を絞るものです。これは便利ですが、守れる範囲はコマンド実行中心です。
>
> 一方で、Dev Container や Docker で Claude Code 自体をコンテナ内で動かすと、Claude Code の実行環境全体をホストから分離できます。
>
> たとえば、ホスト側の `~/.ssh`、`~/.aws`、個人の `.env`、ローカル環境の認証情報などを、コンテナに渡さない設計にできます。
>
> さらに強い隔離が必要な場合は、VMを使う選択肢もあります。
>
> ただし、Dockerなら万能というわけではありません。プロジェクトディレクトリを書き込み可能でmountしていれば、そのコードは変更できます。また、コンテナから外部ネットワークへ出られるなら、情報流出リスクは残ります。
>
> そのため、Dockerを使う場合も、mountする範囲とネットワークの出口、つまりegress設計が重要です。

### 話すポイント

ここでは、Dockerを「魔法の安全装置」と見せないことが重要です。

> **Dockerは被害範囲を小さくする仕組み。秘密情報をmountしたら意味が薄れる。**

---

# 8枚目: 07 実装イメージ

## 最小構成: Dev Container に Claude Code を入れる

**想定時間: 60秒**

### 台本

> 実装イメージとしては、Dev Container に Claude Code を入れて、その中で作業させる構成が分かりやすいです。
>
> `.devcontainer/devcontainer.json` に Claude Code の feature を追加し、Claude Code の設定ディレクトリである `~/.claude` は named volume としてmountします。
>
> こうすると、コンテナをrebuildしても認証情報やClaude Codeの設定を維持できます。
>
> ただし、ここで注意点があります。
>
> まず、`~/.ssh`、`~/.aws`、`.env` を安易にmountしないことです。必要な認証情報は、できるだけ限定された方法で渡します。
>
> 次に、外部通信、つまりegressを絞ることです。必要な接続先だけをallowlistにするのが理想です。
>
> また、`--dangerously-skip-permissions` のように権限確認をスキップする使い方は、通常利用では避けます。使うとしても、承認された隔離環境の中だけに限定します。
>
> 最後に、Dev Container は標準化には便利ですが、それだけで全員に強制できるわけではありません。組織として強制するには、managed settings や端末管理、社内ルールと組み合わせる必要があります。
>
> まとめると、Dev Container は、Claude Code を安全に使うための実行環境テンプレートとして使うのが現実的です。

### コード例の説明

> 画面の例では、Claude Code feature を入れ、`~/.claude` をvolumeにし、不要な通信を抑える環境変数を設定しています。
>
> 実際の社内テンプレートでは、ここに managed settings を組み合わせて、読み取り禁止ファイルや禁止コマンドを定義するのがよいです。

### 話すポイント

このページの一言メッセージです。

> **Dev Container は、Claude Code 用の安全な作業箱として使う。**

## 今日覚えてほしい 3 点

**想定時間: 30〜40秒**

### 台本

> Claude Code の安全運用は、AIを信用するか、信用しないか、という話ではありません。

>

# 時間が足りない場合の短縮版

7分を超えそうな場合は、Docker部分で次の説明だけに絞るとよいです。

> DockerやDev Containerを使う目的は、Claude Code をホストPCから切り離すことです。
> ただし、秘密情報をmountしたり、外部通信を自由にしたりすると、隔離の効果は弱くなります。
> 重要なのは、`~/.ssh`、`~/.aws`、`.env` を渡さないこと、必要な通信先だけ許可すること、そして権限確認を安易にスキップしないことです。

---

# 逆に10分枠で余裕がある場合の補足

10分枠なら、最後に次のような具体例を入れると社員に伝わりやすいです。

> たとえば、Claude Code が「このコマンドを実行してよいですか」と聞いてきたときに、意味が分からなければapproveしないでください。
>
> また、`.env` を見せれば簡単に設定問題を解決できる場合がありますが、それは秘密情報をAIに渡すことになります。代わりに、必要なキー名だけ共有する、ダミー値に置き換える、エラーメッセージだけ共有する、といった方法を使ってください。
>
> つまり、Claude Code に便利に働いてもらう一方で、見せる情報と実行させる操作は人間が管理する、ということです。
