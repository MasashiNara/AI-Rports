Claude Code · Security 10‑min Brief

Claude Code を安全に使うための
3 つの約束

Claude Code はコードを読む・書く・コマンドを実行できる強力な開発エージェントです。だからこそ「レビューする・権限を絞る・隔離環境で動かす」を守って使います。

01なぜセキュリティが必要か0:00–0:45

Claude Code は「チャット」ではなく
開発作業を実行するエージェント

AI に任せる範囲が広がるほど、アクセス権と実行権を絞る必要があります。

コードベースの読み取り、ファイルの書き換え、テストやビルドなどのコマンド実行ができます。便利な反面、誤った変更・秘密情報の読み取り・危険なコマンド実行・外部通信による情報流出といったリスクを伴います。

公式の best practices でも、提案された変更のレビュー、機密リポジトリ向けの権限設定、Dev Container による追加隔離、/permissions の定期確認が推奨されています。

03security-guidance プラグイン1:45–3:00

Claude が書いたコードを、その場でレビュー

  • ファイル編集時危険パターンを高速チェック（モデル呼び出しなし）
  • ターン終了時変更 diff を別 Claude がレビュー
  • commit / push 時周辺コードも含めた深いレビュー
これは自動ブロッカーではない。見逃しもあるため defense‑in‑depth の一層。最後の砦ではなく、早めに気づく安全ネット。
# 公式マーケットプレイスを追加 /plugin marketplace add anthropics/claude-plugins-official # プラグインを導入 /plugin install security-guidance@claude-plugins-official /reload-plugins # 検出対象の例 injection / 危険な deserialize 危険な DOM API / 認可バイパス 安全でない直接オブジェクト参照 / SSRF / 弱い暗号

出典: https://code.claude.com/docs/en/security-guidance

04利用者が守るルール3:00–4:15

開発者が守る 5 つのルール

AI だから特別ではなく、新しい開発者に権限を渡す時と同じ慎重さで扱う。

1Claude の提案をそのまま approve しない
2git diff を見てから承認する
3.env・秘密鍵・認証情報を読ませない
4よく分からない Bash コマンドは許可しない
5機密リポジトリでは sandbox / container を使う
Claude Code はユーザーが承認した範囲だけで動く。だから承認前のレビュー責任は利用者側にもある。
Claude Code Security04 / 07
05社内標準案4:15–5:00

初期設定の推奨案

  • プラグインsecurity-guidance を有効化
  • read 禁止.env / secrets/** は deny
  • 外部通信不明な curl/wget は原則確認
  • 隔離機密リポジトリは Dev Container / Docker
  • 後段PR / CI のレビューは継続
「security-guidance を入れれば安全」とは言わない。in‑session の早期検知層であり、PR・CI・依存スキャンを置き換えるものではない。
// managed-settings.json（抜粋） { "enabledPlugins": { "security-guidance@...": true }, "permissions": { "deny": [ "Read(./.env)", "Read(./secrets/**)", "Bash(curl *)" ], "disableBypassPermissionsMode": "disable" } }

06Docker / Dev Container の意味5:00–6:00

Docker 化の目的：Claude Code をホストから切り離す

方式守れる範囲社員向けの説明
Built‑in Bash sandboxBash コマンド中心コマンド実行の影響を絞る
Dev Container / DockerClaude Code 実行環境全体ホストの秘密情報や環境を直接触らせない
VMOS 全体より強い隔離が必要な場合
Docker でも万能ではない。writable mount のコードは変更でき、外部 egress があれば情報流出リスクは残る。ネットワーク制限と mount 設計が鍵。

07実装イメージ6:00–7:00

最小構成：Dev Container に Claude Code を入れる

// .devcontainer/devcontainer.json { "name": "claude-code-sandbox", "features": { "...devcontainer-features/claude-code:1.0": {} }, "mounts": [ "source=claude-code-config-..., target=/home/vscode/.claude,type=volume" ], "containerEnv": { "CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC": "1" } }
  • 秘密情報~/.ssh・~/.aws・.env を mount しない
  • egress必要な接続先だけを allowlist に
  • bypass--dangerously-skip-permissions は隔離環境のみ
  • 標準化強制には MDM / managed settings が別途必要
~/.claude を named volume で mount すれば rebuild 後も認証維持。Linux では /etc/claude-code/managed-settings.json で組織ポリシーを適用。
